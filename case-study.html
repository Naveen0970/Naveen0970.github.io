Migrating a legacy insurance platform to AEM as a Cloud Service

A migration from legacy CMS platforms to AEM as a Cloud Service — modernizing the component set, content model, authentication, and delivery pipeline to improve governance, modularity, and performance.

Role
Sr. AEM Full Stack Developer
Timeline
Oct 2024 – Nov 2025
Platform
AEM as a Cloud Service
Client
Prudential (Insurance)
01 context

The problem

The platform ran on legacy CMS technology that made content governance and reuse difficult. Authoring was rigid, components were hard to share across sections, and the delivery stack lacked the modular, cloud-native pipeline the team needed to move quickly and safely.

The goal was to migrate onto AEM as a Cloud Service, rebuild content as reusable components and fragments, align authentication with enterprise SAML standards, and stand up an automated Cloud Manager pipeline — all without disrupting live content operations.

Modernize content, security, and delivery on AEMaaCS — while keeping a regulated, content-heavy insurance site stable throughout.
02 architecture

How it fits together

Authors work in AEM, content publishes to the cloud publish tier, and the Dispatcher and CDN cache and secure delivery to end users. Structured content is exposed as Content/Experience Fragments over GraphQL and JSON for headless and SPA channels.

SAML / Azure AD handled authentication, while Content and Experience Fragments fed both the AEM sites and downstream SPA and mobile channels.

03 decisions

Key decisions

A shared component set on editable templates

Rebuilt authoring around editable templates, custom components, and Sling Models in HTL — replacing forked, hard-to-maintain markup with a reusable, policy-driven set authors could configure without developer involvement.

Content & Experience Fragments for headless reuse

Modeled structured content as Content and Experience Fragments and exposed it via GraphQL and JSON, so the same content could serve AEM pages, SPA modules, and mobile channels from a single source of truth.

// Sling Model exposing a Content Fragment to HTL (sanitized)
@Model(adaptables = Resource.class)
public class TeaserModel {

    @ValueMapValue
    private String fragmentPath;

    public String getTitle() {
        return resolve(fragmentPath, "title");
    }
}

SAML auth and a Cloud Manager CI/CD pipeline

Implemented SAML-based authentication with role-based access control to meet enterprise security and compliance needs, and automated builds, tests, and deployments through Cloud Manager, Jenkins, and Maven across environments.

05 takeaways

What I'd carry forward

Migrations succeed on the boring details — a disciplined component model, a clean content structure, and an automated pipeline matter more than any single feature. Treating fragments as the source of truth early made headless and SPA delivery far simpler later, and made the platform easier to govern and extend.
